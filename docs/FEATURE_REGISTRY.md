## How to Use This Doc

- **Feature** — something the user can see or use
- **Editable** — config, copy, tokens, or code that controls behavior
- **Status:** `✓` Done | `○` Partial | `–` Placeholder/TODO | `!` Needs attention

When you ship, migrate, or audit: work through each section and verify nothing was forgotten.

- **`○` (Partial)** — usually API-backed with polish, mock fallbacks, or missing small UI; not “backend missing.”
- **§4 authority** — route modules match [`packages/api/src/server.ts`](../packages/api/src/server.ts); use the **API module index** and bullet subsections for detail; the legacy wide table is not exhaustive.
- **`ecosystem-stubs.ts`** — legacy filename only; handlers are **DB-backed** (see module index). New domains belong in named route modules when possible.

**UX walkthrough (navigation, wayfinding, role affordances, nice-to-haves):** see [`docs/UX_WALKTHROUGH_AUDIT.md`](./UX_WALKTHROUGH_AUDIT.md) — complements this registry’s route table with a page-by-page user lens and a triaged backlog (**Doc-only** / **Small UI** / **Larger UX**).

### Ecosystem map (users ↔ orgs ↔ groups ↔ presenters ↔ vendors ↔ events)

- **Schema (FK summary):** `users` → `profiles`; `organization_members` links `users` to `organizations`; `group_members` links `users` to `groups`; `groups.organization_id` optionally ties a group to an org; `events` references `host_id` (user), `organization_id`, `group_id`; `vendor_profiles` and `presenter_profiles` are keyed by `user_id`; conventions anchor to an org and optionally `anchor_event_id` on `events`; program slots use `schedule_slots` and `schedule_slot_presenters`; **Event Systems** registration uses `convention_registrants.user_id` (unique per convention) + `convention_access_grants` — see [`EVENT_SYSTEMS_IDENTITY.md`](./EVENT_SYSTEMS_IDENTITY.md).
- **Read-model API:** `GET /api/v1/users/:username/ecosystem` — one payload for profile/org discovery UIs (no new parallel tables); includes `**trustScore`** for the Trust ring on public profiles.

### Unified event calendar (simple event + optional convention program)

- **Mental model:** Every public gathering is one `**events`** row. Slots live in `**schedule_slots`**; presenters M2M via `**schedule_slot_presenters**`.

### Organization composite rating + optional global `trustScore` propagation

- **Composite `organizations.rating`:** Blends (1) **weighted star reviews** (org + org-scoped event reviews; long writeups ≥200 chars get extra weight) with (2) **role-weighted `organization_members.local_reputation`** mapped to a 1–5 internal component. See `packages/api/src/lib/org-reputation.ts` and `recalculateOrganizationRating` in `organizations.ts`. `reviewCount` remains the count of star-review rows (not internal rep).
- **Staff deltas:** `PATCH /api/v1/organizations/:orgKey/members/:userId/reputation` applies an integer delta to `localReputation` (MODERATOR+), then recomputes the org composite. Join/leave also triggers recompute.
- **Global profile trust (legacy, off by default):** When `ORG_REVIEW_PROPAGATES_GLOBAL_TRUST=true`, a new org or org-event review still runs `propagateTrustFromOrgReview` (staff get `trustScore` deltas). **Default production template sets this to false** so org reputation stays org-scoped unless you opt in.
- **Conflict of interest:** Users with org role **STAFF** or higher cannot submit org or org-event reviews (API).
- **Attendance (star reviews):** Org-level `**POST .../organizations/:orgKey/reviews`** requires the reviewer to have **attended** at least one **past** org event (RSVP going/maybe). `**POST .../events/:eventId/reviews`** requires **attended** that event. See `packages/api/src/lib/attendance-gate.ts` and `docs/PROJECT_DECISIONS.md`.

## 9. Adding New Features

When you add a feature:

1. Skim `**EXTEND_BEFORE_ADD.md`** and search the repo for existing coverage before adding parallel structures.
2. Register in `packages/api/src/server.ts` and add a row to **§4 API module index** (prefer a bullet subsection over expanding legacy wide rows).
3. Add the route to `router.tsx` and to **Section 2** above.
4. Add the component to **Section 3** if user-facing.
5. Add any new API route to **Section 4** bullets or legacy table if needed.
6. Add mock/state to **Section 5** if applicable.
7. Add a checkbox to **Section 6** for the relevant area.
8. If it introduces new config, add to **Section 1** or **Section 7**.
